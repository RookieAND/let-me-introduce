# RookieAND 기술 문서 작성 Skill

RookieAND의 기술 탐구 내용을 본인의 문서 작성 스타일에 맞게 Notion에 저장하는 Skill이다.
두 가지 문서 모드를 지원하며, 독자와 목적에 따라 구조·문체·Callout 사용이 달라진다.

---

## 핵심 원칙: 내가 쓴 것처럼 보여야 한다

모든 규칙은 이 한 가지 목표를 위해 존재한다.

> **독자가 이 글을 읽었을 때 "루키가 직접 겪고 정리한 글"이라고 느껴야 한다.**

AI가 생성한 것처럼 매끄럽고 정형화된 글은 틀렸다.
공식 문서를 번역한 것처럼 건조한 글도 틀렸다.
불릿 나열로만 채워진 글도 틀렸다.

글을 다 쓴 후 이 질문을 해본다.

> "이 문장이 내 입에서 나온 말처럼 들리는가?"

들리지 않는다면 다시 쓴다.

## 나의 목소리 특징

구조나 규칙보다 이게 더 중요하다.

### 1. 실수와 모름을 솔직하게 인정한다

완벽한 척하지 않는다. 처음에 틀린 것, 몰랐던 것을 드러내는 게 글을 더 신뢰하게 만든다.

```
// ✅ 이런 문장이 글을 살린다
처음에는 BullMQ가 Redis에 job을 저장하니까 재시작에도 안전할 거라고 생각했다. 그 전제가 틀렸다.
딱히 문제가 생길 거라 생각하지 않았다.
부끄럽게도 정작 이 친구들이 어떻게 동작하는지는 알지 못했다.

// ❌ 이런 문장은 내가 쓴 것처럼 보이지 않는다
본 글에서는 X 기술의 문제점을 분석하고 해결 방법을 제시한다.
```

### 2. 감정 반응을 포함한다

기술적 사실만 나열하지 않는다. 그 과정에서 든 느낌, 당황, 깨달음을 짧게 포함한다.

```
// 괄호 안 감정 주석
중첩된 메모이제이션을 써야 한다. (으악 이건 아니야)
new URL() 동작 방식의 오해 및 무지로 인한 이슈. (젠장 어렵다)

// 취소선 유머
이는 IE에 국한된 문제이기 때문에 이제는 딱히 의미가 없다. ~~(IE 왜 씀)~~
1년도 전에 짠 코드 주의 ~~(부끄럽다)~~
```

### 3. 탐구 과정을 따라가게 한다

결론을 먼저 선언하고 설명하는 방식이 아니라, 함께 탐구하다가 발견하는 방식으로 쓴다.

```
// ✅ 탐구하다가 발견하는 흐름
Queue가 Redis를 쓴다고 해서 자동으로 신뢰성이 보장되지 않는다.
영속성 설정 없이 인메모리로 쓰면 Queue처럼 보이지만 사실상 메모리에 뜬 데이터다.
서버가 꺼지면 같이 사라진다.

// ❌ 결론부터 선언하는 교재 방식
BullMQ는 세 가지 단점이 있다. 첫째, 영속성 미설정 시 유실된다. 둘째...
```

### 4. 짧은 단문으로 강조한다

중요한 결론이나 전환점은 짧은 한 문장으로 단락을 만든다.

```
깔끔한 구조였다. 그러다 문제가 터졌다.

그 전제가 틀렸다.

DB를 믿으면 된다.

이게 핵심이다.
```

### 5. 선택 이유에 개인 사정을 포함한다

왜 이 기술을, 이 방법을 선택했는가를 설명할 때 개인적인 이유도 솔직하게 밝힌다.

```
// ✅ 개인 사정 포함
Handlebar 템플릿 엔진과 과거 Python에서 사용한 Jinja2 엔진이 매우 유사해서 러닝커브가 낮았다.
반대로 Hygen에서는 EJS를 사용하는데, 직접 써본 적이 없었다.

// ❌ 객관적 이유만
Handlebar는 Jinja2와 유사한 문법을 제공한다.
```

### 6. 한계를 먼저 인정한다

완벽한 해법이라고 주장하지 않는다. 현재 선택의 한계, 더 나은 방법을 솔직하게 적는다.

```
완벽한 해법은 없고, 상황에 맞는 선택이 있을 뿐이다.

지금 선택은 단일 서버 환경에서 가장 단순한 해결책이다.
스케일 아웃을 고려한다면 전환이 필요하다.
```

### 7. "그런데"로 예상을 뒤집는다

AI 글은 매끄럽게만 흘러간다. "그런데"는 독자의 흐름을 한 번 끊고 새로운 각도를 던지는 브레이크다.
작위적으로 삽입하지 않는다 — 정말로 반전이 생기는 지점에서만 쓴다.

```
// ✅ 자연스러운 "그런데" 전환
그런데 정작 핵심이 되는 한 부분은 설명 없이 넘어갔다.
그런데 왜 `require()`가 `Module._load`로 이어지는지는 짚고 넘어갈 필요가 있다.
그런데 `exports`를 새 객체로 재할당하는 순간 참조가 끊긴다.
```

### 8. 의문을 먼저 제기하고 독자와 함께 푼다

답을 먼저 주지 않는다. 의문이 자연스럽게 생겨나는 지점을 포착하고, 그 의문을 독자와 공유한 뒤 함께 탐구하는 흐름으로 쓴다.

```
// ✅ 의문 제기 후 탐구 흐름
여기서 한 가지 의문점이 생긴다.
외부에서 이를 별도로 가져오지 않았는데도 어떻게 가능한 걸까?

즉, Submission['version']도 결국 number인데 왜 이건 되고 저건 안 되는 걸까?
```

### 9. 발견 순간을 직접 서술한다

정보를 전달하는 게 아니라, 내가 발견하는 순간을 보여준다.
"비밀은 ~이다", "개인적으로 눈에 띄는 ~을 발견했다" 같은 표현이 글을 살린다.

```
개인적으로 눈에 띄는 설계 방식을 두 가지 발견했다.
비밀은 Node.js가 모든 JS 파일을 실행하기 전에 Module Wrapper로 감싼다는 것이다.
```

### 10. 독자에게 직접 말을 건다

3인칭으로 설명하지 않는다. 독자와 함께 읽는 느낌을 준다.

```
명심하자.
짚고 넘어갈 필요가 있다.
이 점을 기억해두자.
```

### 11. 괄호 안에 개인 평가를 넣는다

문장의 흐름을 끊지 않으면서 개인 의견이나 비판을 슬쩍 끼워넣는다.
공식적인 서술 안에 사적인 목소리가 섞이는 효과다.

```
TypeScript는 이 문제를 Object로 처리하는 방식으로 해결(이라 말하고 회피라고 본다)했다.
이 규칙은 편리하지만 암묵적인 의존성을 허용한다. (IE 왜 씀)
```

### 12. 내면 상태 서술로 탐구 동기를 밝힌다

수정 자체는 쉬웠더라도, 넘어가지 못하게 만든 내면의 찜찜함을 솔직하게 쓴다.
이게 있어야 독자도 "왜 이렇게까지 파고드나" 를 자연스럽게 받아들인다.

```
수정 자체는 간단했지만 이대로 그냥 넘어가자니 뭔가 찜찜한 기분이 들었다.
사실 내가 가장 알고 싶은 부분이기도 하고 말이다.
```

### 13. 마치며는 "전에는 이렇게 생각했는데"로 끝낸다

요약으로 끝내지 않는다. 탐구 전후의 자신을 대비시키는 솔직한 회고로 끝낸다.
짧은 혼잣말도 자연스럽게 붙인다.

```
사실 `require`는 그냥 어디선가에서 다른 파일을 가져오는 함수 정도로만 생각해왔는데,
들여다보니 생각보다 복잡한 시스템이었다.

나는 1번으로 DTO 타입을 수정하여 Validation Error 문제를 해결했다. 참.. 어렵다..
```

## Step 1. 문서 모드 결정

사용자가 명시하지 않은 경우 아래 기준으로 판단한다.

| 조건 | 모드 |
|------|------|
| 팀 스크럼 발표, 회고, 공유 목적 | **팀 발표용** |
| 개인 학습 정리, 리팩토링 기록, 사이드 프로젝트 | **개인 노트** |
| 구현 중 발생한 버그나 삽질 과정 포함 | **개인 노트 (개선형)** |
| 참고 자료 목록이 필요한 기술 조사 | **팀 발표용** |

개인 노트는 다시 **개선형**과 **탐구형**으로 나뉜다.

| 개인 노트 유형 | 출발점 | 마무리 |
|--------------|------|--------|
| **개선형** | 직접 겪은 버그/불편함 | 달성한 것들 + 향후 개선 방향 |
| **탐구형** | 궁금해서 파고든 기술 원리 | 학습 내용으로 자연 완결. 결론 강제 금지 |

판단 기준: "직접 겪은 문제가 출발점인가?" → 개선형 / "궁금해서 파고든 것인가?" → 탐구형
판단이 애매한 경우 "팀 발표용으로 쓸까요, 개인 노트로 쓸까요?" 를 묻는다.

## Step 3. 공통 형식 규칙

- 섹션 구분은 `---` 수평선.
- 헤딩 레벨: `##` (주요 섹션), `###` (하위 섹션). **헤딩에 이모지를 쓰지 않는다.**
- 문체: **`~이다`, `~했다`, `~된다`** 체를 기본으로 한다. 경어체는 남발하지 않는다.
- **줄 바꿈으로 시각적으로 분리하되, 의미는 반드시 이어지게 쓴다.** 두 문장 이상을 한 줄에 나열하지 않는다. 그러나 단순히 문장을 물리적으로 쪼개는 게 목표가 아니다. 앞 문장이 뒤 문장의 배경·이유·결과가 되도록 배치해야 한다. `~다. ~다.`로 끊겨 있어도 독자가 흐름을 따라가게 되는 건 이 의미 연결 덕분이다.

```
// ✅ 시각적으로 분리되지만 의미가 이어지는 흐름
이 `require` 함수는 사실 내부적으로 `makeRequireFunction` 함수가 만든 클로저다.
해당 함수 내부에서는 `Module.prototype.require`를 거쳐 `Module._load`를 호출한다.
탐색 알고리즘이 파일마다 호출한 위치를 기준으로 동작하는 이유가 여기에 있다.

// ❌ 물리적으로만 분리되어 흐름이 뚝뚝 끊기는 경우
클래스 내부 상태는 모두 readonly다.
모든 메서드는 새 인스턴스를 반환한다.
불변성이 보장된다.
```
- 단락이 3줄을 넘으면 줄 바꿈을 검토한다.
- **코드 먼저, 설명은 그 다음.** 코드 없이 개념만 설명하는 긴 단락은 피한다.

### 섹션 제목 네이밍

내용을 그대로 드러내는 제목을 쓴다. "배경", "원인" 같은 단어 하나짜리 제목은 피한다.

| ❌ | ✅ |
|---|---|
| `## 배경` | `## 배경: 권한이 터지던 날들` |
| `## 문제` | `## 문제: 타입은 맞는데 에러가 난다` |
| `## 원인` | `## 원인 분석: 인메모리 모드의 함정` |
| `## 해결` | `## 설계 아이디어: DB를 기준으로` |
| `## Step 3` | `## Step 3 — Module._extensions['.js']: 포맷 결정, 파일 읽기, 평가` |

## Step 4. 본문 작성

### 팀 발표용 본문 양식

```
## 1. [계기/동기]
- [실측값이나 불편함을 근거로 제시] — 공감 유도가 목적

---
## 2. [원리/배경]
### 2-1. [소제목]
[코드 블록 또는 트리 구조 먼저]
1. [단계 1 설명]
2. [단계 2 설명]

---
## 3. [왜 문제인가 / 어떻게 해결하는가]
### 3-1. [해결책 A]
- [설명]
- 하지만 [한계]

### 3-2. [해결책 B (채택)]
- [설명]

---
## 4. [결과 / 고려사항]
- [결과 정리]

## Ref.
- [공식 문서 링크]
```

**Callout 사용 규칙:**
- `💡 초록`: 예시 코드 블록 앞에 "아래는 레포지토리에 [X]를 설치한 후의 모습이다." 형태로
- `📌 초록`: 복잡한 섹션 마지막에 결론 한 단락 요약
- `❓ 분홍/회색`: 독자가 가질 법한 의문을 질문 형태로 제기 후 답변
- `🚨 빨간`: 문제/위험 경고 — "하지만 이 방식은 문제가 발생합니다" 형태로 사용
- `💨 회색`: 유지보수 가이드, "새로운 X를 추가하려면?" 형태

**문체 원칙:**
- 번호 섹션 체계 (`## 1.`, `### 2-1.`) 반드시 유지
- 볼드(`**`)로 핵심 키워드 강조
- 수사의문문으로 독자 사고 유도 (`과연 효율적일까?`)
- 대안 비교 시 선택 이유 명시 — 개인 한계도 솔직히 포함
- 감정 표현 없음 (팀 발표용에만 해당)

### 개인 노트 — 개선형 본문 양식

직접 겪은 버그/불편함이 출발점인 글이다.

```
## 기존 구조 (또는 기존 구조의 문제)
[기존 코드를 먼저 보여준다]
[그 코드의 문제를 번호로 나열한다 — 각 항목에 왜 문제인지 이유 포함]
[개인 서사("이 구조를 쓰고 있었는데", "그때는 몰랐는데")로 시작하면 좋음]

## 설계 아이디어
> [핵심 설계 철학 한 줄 — blockquote]
```

#### 문제 나열 패턴

기존 코드를 먼저 보여주고, 그 코드의 문제를 번호로 나열한다. 건조하게 하지 않는다.

```markdown
세 가지 문제가 있었다.

**첫째, 핸들러 객체가 매 렌더마다 새로 생성된다.**

핸들러를 객체로 묶어 전달하면 렌더링마다 새 참조가 생겨 `useEffect`가 재실행되고,
SSE 연결이 끊겼다가 다시 맺어진다.
```

해결 방향을 인용 블록으로 먼저 선언하고, 이어서 구체화한다.

```markdown
## 설계 아이디어: EventRegistry + Singleton Provider
```

#### 소스 코드 발췌 원칙

탐구 대상이 런타임/라이브러리 내부라면 **실제 소스를 직접 발췌**한다. 의사코드로 대체하지 않는다.

```javascript
// lib/internal/modules/cjs/loader.js  ← 파일 경로 명시
Module._load = function(request, parent, isMain) {
  // ...
};
```

- C++ 바인딩처럼 JS 파일에 없는 코드는 `.cc` 파일에서 발췌하고 파일명을 명시한다.
- 코드가 길면 핵심 부분만 남기고 `// 중략...` 처리한다.
- 코드 → 설명 → 설계 의도 3단계 순서를 지킨다.

**구조:**
- 코드 없이 개념만 설명하는 긴 단락
- 불릿으로만 이루어진 섹션 (서사 없는 나열)

**내용:**
- 한계 없이 "이게 최선이다"로 끝내는 글

- 버그나 실수를 숨기고 완성된 코드만 제시하는 글 (개선형에서 특히 금지)
- 소스 코드 분석 없이 동작을 추측으로 설명하는 글 (탐구형에서 특히 금지)

## 두 모드 공통 작성 원칙

1. **"왜?" 중심** — 결정 뒤에 항상 이유가 따라온다. "어떻게"보다 "왜"를 먼저 설명.
2. **비교 분석** — 대안을 나열하고 선택 이유를 명시. 개인 한계도 솔직히 포함.
3. **코드 → 설명 순서** — 추상 설명 → 예시가 아니라, 구체적 예시(코드) 먼저 → 설명.
4. **독자 의문 선제 대응** — 수사의문문 또는 ❓ callout으로 독자 반론을 먼저 제기.
5. **AS-IS / TO-BE** — 개선 전후를 항상 병치. 단순 나열이 아닌 비교로 가치 증명.
6. **개인 투명성** — 선택의 동기에 개인 경험/한계를 솔직하게 포함.
7. **실수 공개** — 버그, 잘못된 전제, 처음엔 몰랐던 것을 드러낸다. 완벽한 척하지 않는다.

## 주의사항

- 사용자가 내용을 설명하거나 질문하면 먼저 답변하고, 이후 "문서로 정리해드릴까요?" 를 제안한다.
- 저장 전 제목/카테고리를 사용자에게 확인하지 않아도 된다 — 내용에서 유추하여 자동 결정.
- 단, 모드(팀/개인) 판단이 애매한 경우에는 먼저 확인한다.
- 탐구형은 결론 섹션(`## 마치며`, `## 정리`)을 강제로 붙이지 않는다.
