# 개발 포스트 작성 가이드라인

---

## 핵심 원칙: 내가 쓴 것처럼 보여야 한다

모든 규칙은 이 한 가지 목표를 위해 존재한다.

> **독자가 이 글을 읽었을 때 "루키가 직접 겪고 정리한 글"이라고 느껴야 한다.**

AI가 생성한 것처럼 매끄럽고 정형화된 글은 틀렸다.
공식 문서를 번역한 것처럼 건조한 글도 틀렸다.
불릿 나열로만 채워진 글도 틀렸다.

글을 다 쓴 후 이 질문을 해본다.

> "이 문장이 내 입에서 나온 말처럼 들리는가?"

들리지 않는다면 다시 쓴다.

---

## 나의 목소리 특징

이 블로그의 글에서 드러나야 하는 특징들이다.
구조나 규칙보다 이게 더 중요하다.

### 1. 실수와 모름을 솔직하게 인정한다

완벽한 척하지 않는다. 처음에 틀린 것, 몰랐던 것을 드러내는 게 글을 더 신뢰하게 만든다.

```
// ✅ 이런 문장이 글을 살린다
처음에는 BullMQ가 Redis에 job을 저장하니까 재시작에도 안전할 거라고 생각했다. 그 전제가 틀렸다.
딱히 문제가 생길 거라 생각하지 않았다.
부끄럽게도 정작 이 친구들이 어떻게 동작하는지는 알지 못했다.
루키의 EJS 미숙으로 인한 선택이기도 했다.

// ❌ 이런 문장은 내가 쓴 것처럼 보이지 않는다
본 글에서는 X 기술의 문제점을 분석하고 해결 방법을 제시한다.
이 방식은 Y보다 우수한 성능을 보여준다.
```

### 2. 감정 반응을 포함한다

기술적 사실만 나열하지 않는다. 그 과정에서 든 느낌, 당황, 깨달음을 짧게 포함한다.

```
// 괄호 안 감정 주석
중첩된 메모이제이션을 써야 한다. (으악 이건 아니야)
new URL() 동작 방식의 오해 및 무지로 인한 이슈. (젠장 어렵다)
이 구조에 꽤나 당황스러웠다.

// 취소선 유머
이는 IE에 국한된 문제이기 때문에 이제는 딱히 의미가 없다. ~~(IE 왜 씀)~~
1년도 전에 짠 코드 주의 ~~(부끄럽다)~~
```

### 3. 탐구 과정을 따라가게 한다

결론을 먼저 선언하고 설명하는 방식이 아니라,
함께 탐구하다가 발견하는 방식으로 쓴다.

```
// ✅ 탐구하다가 발견하는 흐름
Queue가 Redis를 쓴다고 해서 자동으로 신뢰성이 보장되지 않는다.
영속성 설정 없이 인메모리로 쓰면 Queue처럼 보이지만 사실상 메모리에 뜬 데이터다.
서버가 꺼지면 같이 사라진다.

// ❌ 결론부터 선언하는 교재 방식
BullMQ는 세 가지 단점이 있다. 첫째, 영속성 미설정 시 유실된다. 둘째...
```

### 4. 짧은 단문으로 강조한다

중요한 결론이나 전환점은 짧은 한 문장으로 단락을 만든다.

```
깔끔한 구조였다. 그러다 문제가 터졌다.

그 전제가 틀렸다.

DB를 믿으면 된다.

이게 핵심이다.
```

### 5. 선택 이유에 개인 사정을 포함한다

왜 이 기술을, 이 방법을 선택했는가를 설명할 때 개인적인 이유도 솔직하게 밝힌다.

```
// ✅ 개인 사정 포함
Handlebar 템플릿 엔진과 과거 Python에서 사용한 Jinja2 엔진이 매우 유사해서 러닝커브가 낮았다.
반대로 Hygen에서는 EJS를 사용하는데, 직접 써본 적이 없었다.

// ❌ 객관적 이유만
Handlebar는 Jinja2와 유사한 문법을 제공한다.
```

### 6. 한계를 먼저 인정한다

완벽한 해법이라고 주장하지 않는다. 현재 선택의 한계, 더 나은 방법을 솔직하게 적는다.

```
완벽한 해법은 없고, 상황에 맞는 선택이 있을 뿐이다.

지금 선택은 단일 서버 환경에서 가장 단순한 해결책이다.
스케일 아웃을 고려한다면 전환이 필요하다.
```

## 유형 분류

두 가지 유형이 있으며, 목소리 특징은 공통이지만 구조가 다르다.

| 유형 | 핵심 | 예시 |
|------|------|------|
| **문제 해결형** | "내가 겪은 문제와 해결 과정" | `sse-react-event-registry.md`, `casl-abac-declarative-permissions.md` |
| **개발 정보 정리형** | "내가 탐구한 기술과 발견" | `pnpm-dependency-deep-dive.md`, `typescript-enum.md` |

**판단 질문**:
- "직접 겪은 버그/불편함이 출발점인가?" → 문제 해결형
- "궁금해서 파고든 것이 출발점인가?" → 개발 정보 정리형

## 공통 형식

- frontmatter 없음. `##` 헤딩으로 바로 시작한다.
- 섹션 구분은 `---` 수평선.
- 헤딩 레벨: `##` (주요 섹션), `###` (하위 섹션). `#` 사용 금지.
- 문체: **`~이다`, `~했다`, `~된다`** 체. `~입니다`, `~합니다` 절대 금지.
- 단락이 3줄을 넘으면 줄 바꿈을 검토한다.
- **코드 먼저, 설명은 그 다음.** 코드 없이 개념만 설명하는 긴 단락은 피한다.

---

## 문제 해결형 구조

```
## 배경 또는 기존 구조     ← 문제 발생 전 상황. 개인 서사로 시작.
## 발생한 문제             ← 구체적인 버그/불편함. 시나리오 포함.
## 원인 분석               ← "왜?" 중심. 인과관계 서술.
## 설계 아이디어           ← 핵심 아이디어를 > 인용 블록으로 먼저 선언.
## 구현 과정               ← 코드 먼저, 설명은 그 다음.
## 트레이드오프 (선택)     ← 한계, 대안. 솔직하게.
## 인사이트                ← 일반화할 수 있는 교훈.
```

섹션 이름은 내용에 맞게 조정한다. 순서는 유지한다.

**배경 섹션 예시**:
```markdown
## 배경: 권한이 터지던 날들

goorm의 교육 플랫폼에는 강사, 수강생, 운영자 등 수십 가지 역할이 공존한다.
역할마다 접근 가능한 리소스가 달랐고, 같은 리소스라도 읽기만 되는 경우와 수정·삭제까지 되는 경우가 제각각이었다.
```

**코드 변경 비교**:
```typescript
// AS-IS: Queue에 delayed job 등록
await this.evictionQueue.add('auto-evict', { trainingId }, { delay });

// TO-BE: setTimeout으로 직접 등록
setTimeout(() => this.processAutoEviction(training.id), delay);
```

## 개발 정보 정리형 구조

```
## 개념 이름 또는 "~ 란?"     ← 핵심 정의. "왜 공부하게 됐는가" 포함.
## 내부 동작 원리              ← 코드/다이어그램 먼저.
## 상세 개념 (여러 섹션)       ← 각 세부 개념을 순서대로.
## 실전 패턴 (선택)            ← 어떻게 쓰는가.
## 트레이드오프 (선택)         ← 주의사항, 한계.
## 정리                        ← 핵심 요약.
```

**도입부 예시**:
```markdown
## pnpm의 핵심 특징

pnpm이 npm, yarn과 가장 크게 다른 점은 **가상 저장소(CAS)**와 **하드 링크·심볼릭 링크** 조합이다.
```

## 섹션 제목 네이밍

내용을 그대로 드러내는 제목을 쓴다. "배경", "원인" 같은 단어 하나짜리 제목은 피한다.

| ❌ | ✅ |
|---|---|
| `## 배경` | `## 배경: 권한이 터지던 날들` |

| `## 해결` | `## 설계 아이디어: DB를 기준으로` |

## 피해야 할 것

**문체:**
- `~입니다`, `~합니다` 경어체
- "결론적으로", "따라서", "즉" 접속사 남발
- "본 포스트에서는", "이번 글에서는" 같은 관료적 도입

**구조:**
- 코드 없이 개념만 설명하는 긴 단락
- 불릿으로만 이루어진 섹션 (서사 없는 나열)
- 공식 문서를 그대로 번역한 섹션
- 단락을 나누지 않고 여러 문장을 한 블록에 몰아넣기
- `<details>` 접힌 블록 (블로그에서 서사 흐름을 끊는다)

**내용:**
- 한계 없이 "이게 최선이다"로 끝내는 글
- 왜 이 기술/방법을 선택했는지 이유 없이 설명만 하는 글
- 탐구 과정 없이 결론만 있는 글
