# 개발 포스트 작성 가이드라인

---

## 핵심 원칙: 내가 쓴 것처럼 보여야 한다

모든 규칙은 이 한 가지 목표를 위해 존재한다.

> **독자가 이 글을 읽었을 때 "루키가 직접 겪고 정리한 글"이라고 느껴야 한다.**

AI가 생성한 것처럼 매끄럽고 정형화된 글은 틀렸다.
공식 문서를 번역한 것처럼 건조한 글도 틀렸다.
불릿 나열로만 채워진 글도 틀렸다.

글을 다 쓴 후 이 질문을 해본다.

> "이 문장이 내 입에서 나온 말처럼 들리는가?"

들리지 않는다면 다시 쓴다.

---

## 나의 목소리 특징

이 블로그의 글에서 드러나야 하는 특징들이다.
구조나 규칙보다 이게 더 중요하다.

### 1. 실수와 모름을 솔직하게 인정한다

완벽한 척하지 않는다. 처음에 틀린 것, 몰랐던 것을 드러내는 게 글을 더 신뢰하게 만든다.

```
// ✅ 이런 문장이 글을 살린다
처음에는 BullMQ가 Redis에 job을 저장하니까 재시작에도 안전할 거라고 생각했다. 그 전제가 틀렸다.
딱히 문제가 생길 거라 생각하지 않았다.
부끄럽게도 정작 이 친구들이 어떻게 동작하는지는 알지 못했다.
루키의 EJS 미숙으로 인한 선택이기도 했다.

// ❌ 이런 문장은 내가 쓴 것처럼 보이지 않는다
본 글에서는 X 기술의 문제점을 분석하고 해결 방법을 제시한다.
이 방식은 Y보다 우수한 성능을 보여준다.
```

### 2. 감정 반응을 포함한다

기술적 사실만 나열하지 않는다. 그 과정에서 든 느낌, 당황, 깨달음을 짧게 포함한다.

```
// 괄호 안 감정 주석
중첩된 메모이제이션을 써야 한다. (으악 이건 아니야)
new URL() 동작 방식의 오해 및 무지로 인한 이슈. (젠장 어렵다)
이 구조에 꽤나 당황스러웠다.

// 취소선 유머
이는 IE에 국한된 문제이기 때문에 이제는 딱히 의미가 없다. ~~(IE 왜 씀)~~
1년도 전에 짠 코드 주의 ~~(부끄럽다)~~
```

### 3. 탐구 과정을 따라가게 한다

결론을 먼저 선언하고 설명하는 방식이 아니라,
함께 탐구하다가 발견하는 방식으로 쓴다.

```
// ✅ 탐구하다가 발견하는 흐름
Queue가 Redis를 쓴다고 해서 자동으로 신뢰성이 보장되지 않는다.
영속성 설정 없이 인메모리로 쓰면 Queue처럼 보이지만 사실상 메모리에 뜬 데이터다.
서버가 꺼지면 같이 사라진다.

// ❌ 결론부터 선언하는 교재 방식
BullMQ는 세 가지 단점이 있다. 첫째, 영속성 미설정 시 유실된다. 둘째...
```

### 4. 짧은 단문으로 강조한다

중요한 결론이나 전환점은 짧은 한 문장으로 단락을 만든다.

```
깔끔한 구조였다. 그러다 문제가 터졌다.

그 전제가 틀렸다.

DB를 믿으면 된다.

이게 핵심이다.
```

### 5. 선택 이유에 개인 사정을 포함한다

왜 이 기술을, 이 방법을 선택했는가를 설명할 때 개인적인 이유도 솔직하게 밝힌다.

```
// ✅ 개인 사정 포함
Handlebar 템플릿 엔진과 과거 Python에서 사용한 Jinja2 엔진이 매우 유사해서 러닝커브가 낮았다.
반대로 Hygen에서는 EJS를 사용하는데, 직접 써본 적이 없었다.

// ❌ 객관적 이유만
Handlebar는 Jinja2와 유사한 문법을 제공한다.
```

### 6. 한계를 먼저 인정한다

완벽한 해법이라고 주장하지 않는다. 현재 선택의 한계, 더 나은 방법을 솔직하게 적는다.

```
완벽한 해법은 없고, 상황에 맞는 선택이 있을 뿐이다.

지금 선택은 단일 서버 환경에서 가장 단순한 해결책이다.
스케일 아웃을 고려한다면 전환이 필요하다.
```

## 공통 형식

- frontmatter 없음. `##` 헤딩으로 바로 시작한다.
- 섹션 구분은 `---` 수평선.
- 헤딩 레벨: `##` (주요 섹션), `###` (하위 섹션). `#` 사용 금지. **헤딩에 이모지를 쓰지 않는다.**
- 문체: **`~이다`, `~했다`, `~된다`** 체를 기본으로 한다. 경어체(`~입니다`, `~합니다`)는 절대 금지가 아니지만 남발하지 않는다. 주된 문체는 반드시 `~이다` 체여야 한다.
- **두 문장 이상을 한 줄에 나열하지 않는다.** 문장이 끝나면 줄 바꿈으로 구분한다. 읽는 흐름이 끊기지 않도록 문장 단위로 호흡을 나눈다.
- 단락이 3줄을 넘으면 줄 바꿈을 검토한다.
- **코드 먼저, 설명은 그 다음.** 코드 없이 개념만 설명하는 긴 단락은 피한다.

## 개발 정보 정리형

### 구조 템플릿

```
## 도입 또는 핵심 개념 정의
  └─ "왜 이걸 탐구하게 됐는가"를 1~2문장으로. 관련 이전 글이 있으면 연결.
  └─ 핵심 개념을 먼저 정의. 탐구 전체 흐름을 표(단계 요약)로 제시하면 좋음.

## 개념 또는 Step N — 구체 동작 이름
  └─ 소스 코드 발췌 → 동작 설명 → 설계 의도 분석 순서.
  └─ 이전 섹션에서 등장한 함수나 개념이 이 섹션과 연결되면 반드시 언급.

## (반복) 개념 또는 Step N+1
  └─ 앞 섹션의 흐름이 이 섹션으로 어떻게 이어지는지 첫 문장에서 짚는다.

## 보충 개념 (선택)
  └─ 본문 흐름에서 자주 등장하는 내부 패턴(예: primordials)을 별도 섹션으로.

## 마치며 (선택)
  └─ 짧게. 다음 글 예고 또는 핵심 한 줄 요약.
```

### 소스 코드 분석 패턴

탐구 대상이 Node.js / 런타임 / 라이브러리 내부라면 **실제 소스를 직접 발췌**한다.
의사코드나 요약으로 대체하지 않는다.

```javascript
// lib/internal/modules/cjs/loader.js  ← 파일 경로 + 라인 번호(또는 simplified) 명시
Module._load = function(request, parent, isMain) {
  const filename = Module._resolveFilename(request, parent, isMain);
  // ...
};
```

- 소스가 길면 핵심 부분만 남기고 `// 중략...` 처리한다.
- C++ 바인딩처럼 JS 파일에 없는 코드는 `.cc` 파일에서 발췌하고 파일명을 명시한다.
- 코드 블록 아래 **바로** 해석을 붙인다. 코드 → 설명 → 설계 의도 3단계 순서.

### 섹션 간 연결 패턴

각 섹션은 직전 섹션과의 연결을 첫 문장 또는 도입부에서 명시적으로 짚는다.
독자가 앞으로 돌아가지 않아도 흐름을 이해할 수 있어야 한다.

```
// ✅ 이전 섹션 결과를 이어받는 도입
Step 1 에서 `module.load(filename)` 을 호출하면 이 함수가 실행된다.

// ✅ 이전 개념이 현재 섹션의 전제임을 명시
`loadSource` 가 반환한 코드 문자열은 `module._compile` 의 인자로 전달된다.
이 메서드가 코드를 Wrapper Function 으로 감싸는 `wrapSafe` 를 호출하는 진입점이다.

// ❌ 뜬금없이 새 개념을 선언
wrapSafe 함수는 두 가지 경로로 분기된다.
```

### 설계 의도 분석 패턴

코드 설명에서 그치지 않는다.
"왜 이렇게 만들었는가"까지 파고든 흔적을 남긴다.

```
// ✅ 설계 의도까지 분석한 예시
캐시를 실행 전에 미리 저장하는 이유는, 순환 참조 시 무한 루프 없이
"그 시점까지의 exports" 를 안전하게 반환하기 위해서다.

// ✅ 개인적으로 눈에 띈 설계 포인트를 강조
`Module._load` 의 구현체를 분석하던 도중, 개인적으로 눈에 띄는 설계 방식을 두 가지 발견했다.

// ❌ 동작만 서술하고 의도 없음
캐시는 실행 전에 저장된다.
```

### 흐름 전체를 먼저 보여주는 표 패턴

단계가 많은 탐구(예: Node.js 모듈 로딩 파이프라인)는 섹션 진입 전에 전체 흐름 표를 제시한다.
독자가 "지금 어느 단계를 보고 있는지"를 항상 파악할 수 있게 한다.

```markdown
| 단계 | 담당 함수 | 핵심 동작 |
|---|---|---|
| 1 | `Module._load` | 경로 탐색으로 절대경로 결정 |
| 2 | `Module._cache` 확인 | hit → 즉시 반환 / miss → 계속 |
| 3 | `new Module(filename)` | 인스턴스 생성 후 `_cache` 선등록 |
```

## 문제 해결형

### 구조 템플릿

```
## 기존 구조의 문제 (또는 배경)
  └─ 문제 발생 전 상황을 먼저 코드로 보여준다.
  └─ 그 코드가 왜 문제인지를 번호로 나열한다. (3가지 이내)
  └─ 개인 서사("이 구조를 쓰고 있었는데", "그때는 몰랐는데")로 시작하면 좋음.

## 설계 아이디어 (또는 해결 방향)
  └─ 핵심 아이디어를 > 인용 블록으로 먼저 선언.
  └─ 아이디어를 구성하는 컨셉을 불릿으로 정리. (서사 없는 나열은 피하되 여기서는 허용)

## 구현에 필요한 배경 지식 (선택)
  └─ 해결책 이해에 필요한 개념이 있다면 짤막하게 설명.
  └─ "그 전에 X가 뭔지부터 짚고 넘어가자"는 식의 자연스러운 전환으로 시작.

## 구현 과정
  └─ 코드 먼저, 설명은 그 다음.
  └─ 클래스·함수 단위로 ###으로 쪼갠다.
  └─ 설계 결정의 이유("왜 화살표 함수 프로퍼티로 선언했는가")를 반드시 포함.

## 개발 중 만난 버그 (선택, 하지만 강력 권장)
  └─ 실제로 겪은 버그를 ❌/✅ 코드 비교로 서술.
  └─ 원인을 한 문장으로 명확히 설명.

## 달성한 것들 (또는 개선 결과)
  └─ 수치가 있으면 수치로(빌드 시간, 이미지 사이즈).
  └─ 없으면 "단순해진 것", "제거된 것", "보장된 것"을 항목으로.

## 인사이트 (또는 주의사항)
  └─ "이 패턴이 진가를 발휘하는 상황은 이렇다"는 식의 일반화.
  └─ 남용 경고도 포함("반대로 단순한 경우엔 쓰지 않는 게 맞다").
```

### 문제 나열 패턴

기존 코드를 먼저 보여주고, 그 코드의 문제를 번호로 나열한다.
문제 나열은 건조하게 하지 않는다. 각 항목에 **왜** 문제인지의 이유를 포함한다.

```markdown
세 가지 문제가 있었다.

**첫째, 핸들러 객체가 매 렌더마다 새로 생성된다.**

핸들러를 객체로 묶어 전달하면 렌더링마다 새 참조가 생겨 `useEffect`가 재실행되고,
SSE 연결이 끊겼다가 다시 맺어진다.

**둘째, 동일한 Endpoint에 중복 연결이 생긴다.**

Hook 내부에서 독립적인 `EventSource`를 생성하기 때문에,
여러 컴포넌트가 같은 URL로 훅을 호출하면 별도의 Connection이 n개 생긴다.
```

### 설계 아이디어 선언 패턴

해결 방향을 인용 블록으로 먼저 선언하고, 이어서 구체화한다.

```markdown
## 설계 아이디어: EventRegistry + Singleton Provider

> EventSource를 Singleton으로 관리하고, 이벤트가 발생했을 때 등록된 핸들러를 일괄 실행해 재전파한다.

핵심 컨셉은 세 가지다.

- 여러 컴포넌트가 각자 필요한 이벤트 핸들러만 등록한다
- 페이지 내 SSE 연결은 하나만 유지한다
- 서버에서 이벤트가 오면 해당 이벤트를 구독 중인 모든 핸들러에게 전파한다
```

### 배경 지식 삽입 패턴

해결책 이해에 필요한 개념이 있을 때, 별도 섹션으로 짤막하게 설명한다.
분량은 최소화한다. 공식 문서 번역처럼 되면 안 된다.

```markdown
## useSyncExternalStore 란?

그 전에 `useSyncExternalStore`가 뭔지부터 짚고 넘어가자.

`useSyncExternalStore`는 React 18에서 새롭게 선보인 Hook이다.
이름 그대로, **외부 스토어(External Store)의 변경을 React 렌더 사이클에 동기화(Sync)** 한다.
```

### 버그 서술 패턴

개발 중 만난 버그를 숨기지 않는다. **반드시 포함**한다.
버그 이름을 짧게 붙이고, ❌/✅ 코드 비교로 원인을 드러낸다.

```markdown
## 개발 중 만난 버그

**handlers: 0 버그.** 핸들러 등록 시 로그에는 `total: 1`로 찍히는데,
`emit` 시에는 `handlers: 0`으로 나타났다.
원인은 `getHandler()` 메서드에서 Map에 Set을 저장하지 않고 그냥 반환만 했기 때문이다.
매번 새로운 Set이 생성되어 이전 등록이 사라졌다.

// ❌ 잘못된 코드 — 매번 새 Set 생성
// ✅ 올바른 구현
```

### AS-IS / TO-BE 코드 비교 패턴

개선 결과를 보여줄 때는 Before/After 코드를 나란히 제시한다.
수치 개선이 있으면 수치를 함께 제시한다.

```typescript
// AS-IS: setState 콜백 안에 로직이 뒤엉킴
setLearningMap(prev => ({
  ...prev,
  nodes: prev.nodes.map(...)
}));
```
