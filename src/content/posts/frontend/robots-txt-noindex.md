## 배경: CS 채널에서 날아온 요청

최근 CS 로 특정 채널에 속한 페이지 일부를 검색 엔진에 잡히지 않게 수정해달라는 요청을 받았다.  
비즈니스 관점에서의 이야기라서 자세한 내용은 생략하지만, 요구 사항을 정리하면 크게 두 가지가 있었다.

1. 이미 검색 엔진에 노출된 페이지들을 전부 내리기
2. 새롭게 페이지들이 검색 엔진에 잡히지 않도록 처리

따라서 작업 순서는 신규 크롤링 차단을 먼저 처리한 뒤, 기존 색인 제거를 그 다음으로 잡았다.  
매우 급하게 처리해야 하는 상황이었지만, 처음에는 그렇게 복잡한 작업은 아닐 거라고 생각했다.

---

## 처음 떠올린 방법 — robots.txt

이번 CS 를 받자마자 든 첫 생각은 `robots.txt` 을 수정하는 것이었다.

> 페이지 index 마다 다른 robots.txt 를 반환하면 되지 않을까?

페이지 별로 크롤링 허용 여부가 달라야 하기 때문에 요청이 들어오는 경로에 따라  
동적으로 `robots.txt` 를 생성하는 미들웨어를 빠르게 작성해서 끼워 넣었다.  
여기까지는 계획대로 순조로웠다.

그런데 `robots.txt` 의 정확한 작성법을 다시 한번 확인하려고 구글 공식 문서를 뒤지다가 `noindex` 라는 키워드를 발견했다.  
이미 `robots.txt` 로 크롤링은 차단해뒀지만, 혹시 `noindex` 도 함께 걸어두어야 더 확실하게 막을 수 있는 건 아닐까 싶었다.  
그래서 둘 다 처리해두기로 마음먹었는데, 막상 적용하려고 보니 두 방식이 실제로 어떻게 다른지부터 제대로 이해하고 넘어가야 할 것 같았다.

---

## robots.txt 와 noindex 는 어떻게 다른가

둘 다 검색 엔진이 페이지를 처리하는 방식에 영향을 주지만, 동작하는 레이어가 다르다.

- **robots.txt** — 크롤러가 특정 페이지에 **접근하는 것 자체**를 차단한다.
- **noindex** — 크롤러는 페이지에 접근할 수 있지만, **검색 결과에 노출하지 않도록** 지시한다.

접근 차단과 색인 제외는 서로 다른 레이어의 개념이다.  
그러면 둘 다 걸어두면 더 확실하지 않을까 생각했는데, 구글 공식 문서를 꼼꼼히 읽다가 두 가지 경고를 마주쳤다.

**robots.txt 문서:**

> robots.txt 파일을 Google 검색결과에서 웹페이지를 숨기는 수단으로 사용하지 마세요.  
> 다른 페이지 내의 설명 텍스트에서 내 페이지를 가리키는 경우 Google은 페이지를 방문하지 않고도 URL의 색인을 생성할 수 있습니다.

**noindex 문서:**

> `noindex` 규칙이 효과적으로 작동하려면 페이지 또는 리소스가 robots.txt 파일로 차단되어서는 **안 되며** 크롤러에서 페이지에 액세스할 수 있어야 합니다.  
> robots.txt 파일로 페이지가 차단되거나 크롤러가 페이지에 액세스할 수 없다면 크롤러는 `noindex` 규칙을 **절대 확인하지 않습니다.**

이 두 경고를 동시에 읽으면서 느끼는 점은, 두 조건이 서로 상호 충돌을 일으키는 존재라는 것이었다.

---

## robots.txt + noindex 를 함께 쓰면 생기는 문제

두 문서를 합쳐서 생각해보면 아래 상황이 발생한다.  
크롤러가 접근 가능한 다른 페이지에서 차단된 페이지의 링크를 발견했다는 상황을 가정해보자.

1. 페이지 내 URL 이 존재하기에 색인이 생성된다 (크롤러가 아직 페이지를 방문하진 않았음)
2. 그런데 robots.txt 가 해당 페이지 접근을 차단하고 있어, 크롤러가 noindex 태그를 읽으러 들어갈 수 없다
3. 결국 noindex 설정 자체를 크롤러가 인지하지 못하여 메타 정보 없는 URL 이 검색 결과에 영원히 노출됨

즉 색인은 생성되는데 noindex 는 작동하지 않는 최악의 상태가 된다. (으악 이건 아니야)

그리고 여기서 한 가지 더 알게 된 것이 있다면, **색인 생성은 페이지 방문 없이도 일어난다는 점이다.**

크롤러가 직접 페이지에 들어오지 않아도, 다른 페이지에서 링크가 걸렸다면 URL 이 색인에 등록될 수 있다.  
검색 결과에 내용이 채워지는 건 크롤러가 페이지에 들어온 이후의 이야기고, 색인 등록은 그보다 먼저 발생한다.  
나는 robots.txt 로 막아두면 URL 자체가 색인에 등록되는 것도 막을 수 있다고 생각했는데, 그 전제가 틀렸다.

---

## 올바른 방법

> 검색 엔진에서 제외하고 싶으면 robots.txt 로는 막지 않아야 하며 noindex 만 넣어야 한다.

왜냐하면 크롤러가 페이지에 접근할 수 있어야 noindex 를 읽고 색인에서 제외할 수 있다.  
따라서 처음에 만든 동적 robots.txt 미들웨어는 해당 채널 경로에 대해 접근을 허용하는 방향으로 수정하고,  
각 페이지 별 `<head>` 에 noindex 메타 태그를 심는 방식으로 마무리했다.

```html
<meta name="robots" content="noindex, nofollow" />
```

## 이미 색인된 페이지는 어떻게 내리나

정말 다행히도 Google Search Console 에서 제거 요청을 할 수 있다. 처리 속도는 거의 즉각적인 느낌이었다.

단, 한번 제거된 페이지 색인은 6개월 뒤에 다시 색인될 수 있다는 점을 주의해야 한다.  
Search Console 제거 요청은 어디까지나 임시 조치이니 그 전에 반드시 noindex 처리를 마쳐두어야 한다.
