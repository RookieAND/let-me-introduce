## 크롬 개발자 도구를 제대로 쓰고 있었는가

개발자 도구를 쓰긴 했지만, Elements·Console·Network 탭 이외는 거의 열어보지 않았다.
Memory 탭이 있다는 건 알았는데 어떻게 쓰는지 몰라서 피했다.
스터디하면서 제대로 정리할 기회가 됐다.

디버깅할 때는 시크릿 모드를 켜는 게 좋다.
브라우저 확장 프로그램이나 캐시가 결과에 영향을 줄 수 있기 때문이다.

---

## Elements 탭

웹 페이지의 HTML과 CSS를 확인하고 직접 수정할 수 있다.

요소를 선택하면 크기 정보가 색상 코드로 표시된다.

- 파랑: 실제 너비·높이
- 초록: 패딩
- 주황: 마진
- 노랑: 보더

오른쪽 패널에서 선택 요소의 상세 정보를 볼 수 있다.

- **Computed**: 계산된 CSS 최종 값
- **Event Listener**: 해당 요소에 직접 등록된 이벤트 리스너 목록 (이벤트 버블링으로 올라온 것은 표시 안 됨)

---

## Sources 탭

애플리케이션에서 사용하는 모든 파일을 열람할 수 있다.

Production 모드에서는 파일이 번들링되어 읽기 어렵다.
개발 모드에서는 소스맵 덕분에 원본 모듈 파일을 추적할 수 있다.

코드에 중단점을 걸어 JS 실행을 멈추고 디버깅을 진행할 수 있다.

---

## Network 탭

페이지 접속부터 발생하는 모든 네트워크 요청이 기록된다.

XHR, Fetch, WS 등 요청 종류별 필터링이 가능하다.
스크린샷 캡처 기능으로 특정 타임라인의 네트워크 흐름을 시각적으로 확인할 수 있다.

확인해야 할 항목은 다음과 같다.

- 중복되거나 불필요한 요청이 있는지
- 리소스 크기가 과도하게 크지 않은지
- 특정 리소스 로딩이 병목을 만들지 않는지
- preload 순서가 올바른지

모바일 웹을 만들 때는 리소스 크기에 더 신경 써야 한다.
사용자가 데이터를 소비하는 비용이 직접 돈으로 연결되기 때문이다.
gzip이나 brotli 압축을 잘 활용하는 것이 좋다.

## Memory 탭

메모리 누수, 성능 저하, 프리징 현상을 진단하는 탭이다.

세 가지 프로파일링 도구를 제공한다.

### Heap Snapshot

현재 시점의 메모리 상태를 촬영한다.
특정 시점에 어떤 객체가 메모리에 있는지 파악할 수 있다.

두 스냅샷을 비교할 때는 "스냅샷 1에서 스냅샷 2 사이에 할당된 객체"를 선택하고
"얕은 크기" 기준으로 내림차순 정렬하면 새로 추가된 객체를 추적하기 좋다.

- **얕은 크기**: 객체 자체가 보유한 메모리 크기
- **유지된 크기**: 객체와 그 객체가 참조하는 모든 자식 객체를 합산한 크기

두 값의 차이가 클수록 해당 객체가 많은 것을 참조하고 있다는 뜻이다.
메모리 누수 의심 시 이 차이가 큰 객체를 우선 확인한다.

메모이제이션이 제대로 됐는지 확인할 때도 유용하다.
두 스냅샷 비교에서 메모이제이션된 함수나 값이 나타나지 않으면 재생성이 없다는 뜻이다.

### 타임라인 할당 계측

시간에 따른 메모리 변화를 연속으로 관찰한다.
Heap Snapshot보다 메모리 부담이 크지만, 어떤 인터렉션이 메모리를 증가시키는지 추적하기 좋다.

특정 변수를 전역 변수로 저장하면 Console에서 값을 직접 확인할 수 있다.

### 할당 샘플링

JS 함수들을 실행 스택 별로 열람한다.
무거운 순으로 정렬하면 메모리를 가장 많이 점유하는 함수를 찾을 수 있다.
소스 패널 표시를 켜면 해당 함수가 어느 모듈에서 정의됐는지 확인 가능하다.

## Next.js 서버 디버깅

Next.js를 디버그 모드로 실행하면 Node.js 디버거가 활성화된다.

```bash
"dev": NODE_OPTIONS='--inspect' next dev
```

이후 Chrome 브라우저에서 `chrome://inspect`로 이동해 "Open Dedicated DevTools for Node"를 클릭한다.

부하 테스트가 필요하면 `ab`(ApacheBench)를 활용한다.

```bash
# 동시 10개 요청, 총 100번 실행
ab -k -c 10 -n 100 http://localhost:3000/
```

응답 시간, Byte 크기 같은 정보를 통해 서버 성능을 측정할 수 있다.
