## Module Federation이란?

Module Federation(MF)은 JavaScript 애플리케이션을 분산화하기 위한 아키텍처 패턴이다. 서버 사이드의 마이크로서비스와 유사한 개념을 프론트엔드에 적용한 것으로, **하나의 빌드 산출물에서 다른 빌드 산출물의 코드를 런타임에 동적으로 로드하고 공유**할 수 있도록 해준다.  

Webpack 5에서 처음 도입됐으며, 현재는 Rspack, Vite, Metro 등 다양한 빌드 도구를 지원한다. Module Federation 2.0에서는 동적 타입 힌팅, Manifest, Federation Runtime, Runtime Plugin System 등의 기능이 추가됐다.  

---

## 핵심 용어

**Producer (Remote)**: `exposes` 설정을 통해 모듈을 외부에 노출하는 애플리케이션이다.  

**Consumer (Host)**: `remotes` 설정을 통해 다른 애플리케이션의 모듈을 가져다 사용하는 애플리케이션이다.  

**Manifest** (`mf-manifest.json`): Producer가 빌드 시 생성하는 런타임 명세 파일이다. 노출된 모듈 목록, 에셋, 공유 의존성, 타입 파일 위치 등을 기술한다.  

**Shared**: 여러 애플리케이션 간에 공유되는 의존성이다. 동일한 라이브러리가 중복 로딩되지 않도록 버전 협상을 통해 재사용된다.  

**Share Scope**: Shared 의존성이 등록되는 풀(pool)의 이름이다. 기본값은 `default`이며, 복수의 스코프를 사용해 의존성을 격리할 수 있다.  

**Bridge**: 서로 다른 프레임워크로 개발된 완전한 애플리케이션을 모듈로 로드·통합하기 위한 솔루션이다. 컴포넌트 수준이 아닌 애플리케이션 수준의 통합을 담당한다.  

---

## MF로 얻을 수 있는 것

**팀 간 독립 개발 및 배포.** 각 팀이 자신의 애플리케이션을 독립적으로 개발하고 배포할 수 있다. 한 팀의 배포가 다른 팀의 빌드·배포 사이클에 영향을 주지 않는다.  

**공유 의존성 최적화.** React, antd 등 공통 라이브러리를 Shared로 설정하면 중복 번들링이 제거된다. Shared Tree Shaking을 적용하면 실제로 사용하는 모듈만 번들에 포함된다.  

**애플리케이션 수준의 통합 (Bridge).**  

라우팅과 내부 상태 관리를 포함한 애플리케이션을 통째로 다른 앱에 마운트할 수 있다.  

서로 다른 프레임워크(React ↔ Vue 등)로 개발된 앱 간의 통합도 가능하다.  

**점진적 마이그레이션.** 기존 모놀리식 애플리케이션을 한 번에 교체하지 않고, 기능 단위로 점진적으로 분리할 수 있다.

## 도입 시 고려 사항

### 빌드 도구 호환성

MF는 Webpack 5, Rspack, Rsbuild, Vite 등을 지원하나, 빌드 도구별로 제공되는 플러그인과 설정 방식이 다르다. 현재 프로젝트의 빌드 도구가 MF 플러그인을 공식 지원하는지 먼저 확인해야 한다.  

### Shared 의존성 전략

`singleton: true`로 설정된 라이브러리(예: React)는 페이지에 단 하나의 인스턴스만 존재해야 한다.  

버전 불일치 시 충돌이 발생할 수 있다.  

여러 팀이 동일한 라이브러리의 서로 다른 버전을 사용할 경우, `requiredVersion`, `strictVersion` 같은 버전 협상 전략을 명확히 정의해야 한다.  

> [!WARNING]  
> `singleton: true` 라이브러리의 버전이 Producer와 Consumer 간에 불일치하면 런타임 에러가 발생한다. 빌드 타임에는 에러가 없으므로, 배포 후에야 문제가 드러난다.  

### CSS 격리

MF는 CSS 스타일 격리를 기본 제공하지 않는다. CSS Modules, BEM, CSS-in-JS 등 격리 전략을 팀 차원에서 사전에 합의해야 한다. Shadow DOM 방식은 컴포넌트 라이브러리와의 호환성 문제가 생길 수 있어 신중하게 검토해야 한다.  

### 타입 안전성

MF 2.0은 원격 모듈의 동적 타입 힌팅을 지원한다. `tsconfig.json`에 경로 설정을 추가하면 원격 모듈을 로컬 모듈처럼 타입 추론할 수 있다. 타입 파일은 빌드 시 자동 생성되며, 개발 중에는 핫 리로딩도 지원된다.  

### 운영 복잡도

여러 애플리케이션이 런타임에 서로 의존하므로, 하나의 Producer 배포가 Consumer에 즉시 영향을 준다. Manifest를 활용한 버전 관리와 디버깅 환경을 미리 구축해두는 것이 좋다. `errorLoadRemote` 훅을 통한 폴백 처리를 반드시 고려해야 한다.  

---

## 정리

MF는 대규모 팀이 독립적으로 개발하고 배포해야 하는 환경에서 강력한 도구다. 하지만 CSS 격리, Shared 의존성 충돌, 운영 복잡도 등 해결해야 할 문제들이 있다. 도입 전에 팀 차원에서 전략을 충분히 논의하고 프로토타입으로 검증하는 과정이 필요하다.
