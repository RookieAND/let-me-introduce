## 성능은 측정할 수 있어야 개선이 된다

웹 성능을 개선한다는 말은 막연하게 들린다.  
무엇을 기준으로 "빠르다", "느리다"를 판단하는가?  

Google이 Core Web Vitals를 제시한 건 이 질문에 대한 답이다.  
사용자 경험을 수치로 측정하고, 개선 목표를 명확하게 설정할 수 있다.  

로딩이 1초 지연될 때마다 전환률이 4.42% 감소한다는 데이터도 있다.  
개발 환경에서 빠르다고 해서 운영 환경에서도 빠른 게 아니다.  
실제 사용자 환경에서의 측정이 필요한 이유다.

## LCP — Largest Contentful Paint

뷰포트에서 가장 큰 이미지나 텍스트 블록이 렌더링되기까지 걸리는 시간이다.  

**목표**: 2.5초 이내 `Good`, 4.0초 초과 `Poor`  

LCP에 영향을 주는 요소는 주로 이미지다.  
히어로 이미지, 배경 이미지, 큰 텍스트 블록이 LCP 대상이 된다.  

개선 방법은 다음과 같다.  

- 이미지 크기 최적화 (WebP, AVIF 포맷)
- 배경 이미지 대신 `<img>` 태그 사용 (프리로드가 가능하므로)
- LCP 이미지에 `<link rel="preload">` 추가
- CDN 활용으로 서버 응답 시간 단축

## FID — First Input Delay

사용자가 처음 상호작용했을 때부터 브라우저가 응답하기까지 걸리는 시간이다.  

**목표**: 100ms 이하 `Good`, 300ms 초과 `Poor`  

Long Task가 주요 원인이다.  
메인 스레드를 50ms 이상 블록하는 작업이 있으면 FID가 나빠진다.  

2024년부터 FID 대신 **INP(Interaction to Next Paint)**가 핵심 지표로 채택됐다.  
FID는 첫 입력만 측정하지만, INP는 페이지 전체 수명 동안의 모든 상호작용 응답을 측정한다.  

개선 방법은 다음과 같다.  

- 작업을 작은 단위로 분리 (setTimeout, requestIdleCallback)
- 코드 스플리팅으로 초기 번들 크기 축소
- 무거운 라이브러리 lazy 로딩

## CLS — Cumulative Layout Shift

사용자가 예상하지 못한 레이아웃 변경이 얼마나 발생했는지 측정한다.  

**목표**: 0.1 이하 `Good`, 0.25 초과 `Poor`  

읽고 있던 텍스트가 갑자기 아래로 밀리거나, 버튼 위치가 바뀌어 잘못 클릭하는 경험이 CLS 문제다.  

원인과 해결 방법은 다음과 같다.  

- **이미지에 `width`·`height` 명시**: 브라우저가 공간을 미리 확보한다
- **폰트 FOUT 방지**: `font-display: optional` 또는 폰트 Preload
- **동적 콘텐츠 공간 예약**: 광고나 배너가 들어올 자리를 미리 지정
- **transform 애니메이션 사용**: `top`·`left` 변경은 Reflow를 유발하지만 `transform`은 그렇지 않다

## 추가 지표

**TTFB (Time to First Byte)**  

요청 후 첫 번째 바이트가 도착하기까지 걸리는 시간이다.  
600ms 이하를 권장한다.  
서버 응답 속도, CDN 적용 여부가 직접적인 영향을 준다.  

**FCP (First Contentful Paint)**  

첫 번째 DOM 요소(텍스트, 이미지, 캔버스 등)가 화면에 나타나는 시간이다.  
1.8초 이내를 권장한다.

## RAIL 모델

Google이 제시한 성능 목표 프레임워크다.  

- **Response**: 사용자 입력에 100ms 이내 응답
- **Animation**: 16ms 이내 프레임 (60fps 유지)
- **Idle**: 유휴 시간에 50ms 단위로 작업 분할
- **Load**: 5초 이내 인터렉션 가능 상태 도달

RAIL 모델은 어떤 상황에서 어느 지표를 최적화해야 하는지 우선순위를 잡는 데 도움이 된다.
